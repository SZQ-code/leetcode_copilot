# 深色 IDE 全站前端升级设计

- 日期：2026-06-29
- 状态：已确认
- 范围：`frontend/` 全站视觉、布局与交互结构
- 视觉方向：B 方案，固定深色 IDE 专注台
- 布局方向：A 方案，持久工作区外壳
- 优先级：项目展示效果 > 桌面端刷题效率 > 移动端体验

## 背景

LeetCode Copilot 已具备题目解析、历史记录、题型归纳、学习状态管理和 Tutor Agent 等完整功能。当前前端采用浅色横向导航和通用卡片布局，基础可用，但存在以下问题：

1. 首页主要展示开发状态，没有充分呈现真实学习能力和产品价值。
2. 各页面缺少统一的工作区结构，页面之间更像独立功能页。
3. 解题、题解和 Agent 是核心体验，但视觉层级不足。
4. 移动端导航会换行，页面头部占用较多纵向空间。
5. 状态、加载、错误和空数据反馈在不同页面之间不够统一。
6. 现有视觉辨识度有限，不利于项目展示。

本次升级在不改变后端接口和业务范围的前提下，重组前端信息结构并建立统一的深色 IDE 视觉系统。

## 目标

1. 让全站形成统一、可识别的算法学习工作区。
2. 强化项目展示效果，使首页和核心页面能够直接表达产品能力。
3. 提高桌面端输入题目、阅读解析、复盘记录和使用 Tutor Agent 的效率。
4. 保持全部现有路由、业务功能和后端数据流。
5. 修复移动端导航换行和双栏内容拥挤问题。
6. 统一加载、空数据、错误、保存和 Agent 工具状态。
7. 保持键盘可用性、可读对比度和减少动态效果支持。

## 非目标

- 不修改 FastAPI 路由、数据库模型或 API 响应结构。
- 不增加账号、主题切换、搜索、通知或其他新业务功能。
- 不引入全局状态管理库。
- 不引入远程字体、图标包或大型 UI 组件库。
- 不把所有页面强制改造成相同的双栏结构。
- 不重写现有题目、分类或 Agent 请求逻辑。
- 不在本阶段引入新的前端测试框架。

## 方案选择

### 视觉方向

采用固定深色 IDE 专注台。

未采用的方向：

1. 算法实验手册：学习感清晰、长期耐看，但项目展示冲击力弱于深色 IDE。
2. 学习地图仪表盘：成长反馈突出，但当前数据量较少时首页容易显空。

固定深色主题能够建立明确的开发工具气质，并避免本阶段扩大为双主题组件系统。

### 布局方向

采用持久工作区外壳。

未采用的方向：

1. 顶部导航工作台：改造风险低，但更接近视觉换肤，品牌辨识度不足。
2. 全局双栏控制台：Agent 存在感强，但首页、历史和题型页面会受限，移动端适配成本过高。

持久工作区外壳兼顾展示辨识度、桌面操作效率和各页面独立组织内容的需要。

## 全站结构

### 桌面端

`1024px` 及以上使用三层工作区：

```text
┌────────┬───────────────────────────────────────────┐
│        │ Context Bar / 当前模块 / 服务状态         │
│ Side   ├───────────────────────────────────────────┤
│ Rail   │                                           │
│        │ Main Workspace                            │
│        │                                           │
└────────┴───────────────────────────────────────────┘
```

- 左侧持久导航栏承载首页、开始解题、刷题记录和题型归纳。
- 顶部上下文状态栏显示当前模块、必要的页面操作和后端连接状态。
- 主工作区根据页面用途使用单栏、双栏或数据网格。
- 传统页脚移除，项目标识和服务状态并入工作区外壳。

### 平板端

`768–1023px` 保留紧凑图标导航栏，隐藏辅助导航文字，并缩小内容区边距。

### 移动端

小于 `768px` 时：

- 左侧导航栏替换为顶部品牌栏和底部主导航。
- 双栏内容改为单栏堆叠或已有标签切换。
- 页面不产生横向滚动。
- 底部导航为正文内容预留安全间距。

## 页面设计

### 首页

首页从开发状态页改为产品能力入口：

1. 首屏用真实产品文案说明“题目输入—AI 解析—Agent 追问—学习复盘”的完整闭环。
2. “开始一道题”是唯一主操作，“查看复习队列”作为次操作。
3. 使用现有 `GET /api/problems` 展示最近学习记录。
4. 使用现有 `GET /api/categories` 展示待复习数量或薄弱题型摘要。
5. 数据请求失败时保留首屏和主操作，只在对应数据区域显示错误。
6. 没有学习数据时使用明确的首次解题引导，不显示空白统计面板。

### 开始解题

- 桌面端使用输入区和解析区双栏布局，比例约为 `44/56`。
- 输入区保留题目文本、字符数量、Provider 说明和主按钮。
- 解析区在空闲、加载、错误和成功状态之间保持稳定尺寸，减少布局跳动。
- 成功后突出已保存记录编号和详情入口。
- 移动端改为输入区在上、解析区在下。

### 刷题记录

- 页面标题、说明和记录总览形成紧凑工具栏。
- 记录使用高密度列表或卡片行，清晰区分标题、难度、掌握状态、标签和时间。
- 最新记录保持在最前面。
- 加载、错误和首次使用空状态采用统一状态组件。
- 本阶段不新增搜索、排序或筛选功能。

### 题型归纳

- 顶部展示总体掌握情况和复习数量。
- 中部题型卡片继续支持标签选择。
- 下部复习队列继续支持“待复习 / 全部记录”切换。
- 被选题型、低掌握率和待复习状态使用语义色，而不是仅依赖文字。
- 保持现有两个独立请求及其独立错误边界。

### 题目详情

- 题解和学习记录构成主工作区。
- Tutor Agent 在桌面端固定于右侧，维持当前独立滚动和工具轨迹。
- 原题、学习记录和题解按阅读顺序组织，避免多个同权重卡片争夺注意力。
- 保存状态在学习记录面板内部反馈。
- Agent 消息、工具确认和错误在 Agent 面板内部反馈。
- 移动端继续使用“题解 / 题目导师”标签切换。

### 404 页面

- 纳入统一工作区外壳。
- 使用明确的路由错误说明和返回首页操作。
- 不显示不存在的服务状态或开发术语。

## 视觉系统

### 颜色

| Token | 色值 | 用途 |
| --- | --- | --- |
| `void` | `#0B0F16` | 页面背景 |
| `shell` | `#111721` | 导航与外壳 |
| `panel` | `#151D29` | 内容面板 |
| `panel-raised` | `#1A2330` | 强调面板 |
| `line` | `#2B3546` | 边框与分隔 |
| `text` | `#F2F5FA` | 主文字 |
| `muted` | `#8D9AAF` | 辅助文字 |
| `cyan` | `#79D8FF` | 主操作、当前状态 |
| `green` | `#7DDE9B` | 成功、在线 |
| `amber` | `#E8BF67` | 复习、提醒 |
| `red` | `#FF7A86` | 错误、危险状态 |

颜色不能单独承担状态含义，必须同时提供文字、图标或形状差异。

### 字体

- 展示标题：`Bahnschrift Condensed`，回退到 `Arial Narrow` 和 `Segoe UI`。
- 正文与控件：`Segoe UI Variable`，回退到 `Segoe UI`。
- 代码、状态和小标签：`Cascadia Code`，回退到 `Consolas`。

所有字体使用本地系统字体，不发起远程字体请求。

### 形状与层级

- 内容面板使用 `6–10px` 小圆角。
- 通过背景明度、薄边框和有限阴影建立层级。
- 不使用大面积玻璃模糊、霓虹外发光或无语义渐变。
- 主要按钮使用实心青色，次要按钮使用边框样式，危险操作使用红色。
- 标签和状态采用紧凑的等宽字体样式。

### 标志性元素：执行轨迹

执行轨迹只用于存在真实顺序或状态变化的内容：

- 题目输入、解析、保存和复盘流程。
- Agent 工具调用及确认状态。
- 加载、成功和错误状态转换。

执行轨迹由细线、状态节点和短标签组成，不作为无意义装饰，也不在普通并列卡片中滥用编号。

### 动效

- 导航、按钮、面板展开和状态更新使用 `160–220ms` 过渡。
- 页面不使用持续背景动画。
- 后端连接状态可使用低频状态变化，但不能持续闪烁。
- `prefers-reduced-motion: reduce` 时关闭非必要动画和滚动行为。

## 组件边界

### `AppShell`

职责：

- 渲染桌面侧栏、平板紧凑栏、移动顶部栏和底部导航。
- 根据当前路由生成上下文标签。
- 调用现有 `GET /health` 显示后端连接状态。
- 为主工作区提供统一宽度、间距和底部安全区。

`AppShell` 不管理题目、分类或 Agent 业务状态。

### 导航组件

- 导航数据只维护一份。
- 图标使用当前代码库内的内联 SVG。
- 当前路由同时通过颜色、背景和左侧轨迹线标识。
- 移动底部导航提供文字标签，避免只有图标。

### 状态组件

建立共享的加载、空数据、错误和成功提示组件：

- 加载状态说明正在执行的动作。
- 空数据状态提供下一步操作。
- 错误状态说明失败原因和可执行的恢复方式。
- 成功状态使用与触发操作一致的动词。

业务页面继续决定何时显示状态，共享组件不发起请求。

### 内容组件

现有 `ProblemInput`、`SolutionPanel`、`ProblemCard`、`LearningOverview`、`CategoryStatCard`、`LearningRecordPanel` 和 `TutorAgentPanel` 保留职责，通过统一 Token 和必要的内部结构调整完成视觉升级。

不在本阶段将所有页面合并为单个大型组件，也不引入按视觉命名但缺乏业务含义的组件层。

## 数据流

- 页面继续管理自己的加载、成功和错误状态。
- `frontend/src/api/problems.ts` 继续作为 API 访问入口，并增加健康检查函数。
- 首页复用现有题目和分类接口，不新增后端请求类型。
- 解题页继续使用 `idle → loading → success/error` 状态流。
- 题型页继续维护统计与题目列表两个独立请求。
- 详情页继续独立管理题目记录、保存状态和 Agent 会话。
- 不使用 Context 保存业务数据；`AppShell` 仅处理布局和全局导航状态。

## 错误处理

| 场景 | 界面行为 |
| --- | --- |
| 后端健康检查失败 | 顶部状态显示“后端未连接”，页面已有数据和静态入口继续可用 |
| 首页最近记录失败 | 最近记录区域显示局部错误，首屏和解题入口保持可用 |
| 解题请求失败 | 解析区显示具体错误，输入内容保持不变 |
| 历史记录失败 | 记录区域显示错误和后端检查提示 |
| 分类统计失败 | 统计区域显示错误，不伪造空数据 |
| 复习队列失败 | 队列区域独立显示错误，已加载统计继续展示 |
| 详情加载失败 | 显示记录不存在或连接失败，并提供返回记录列表操作 |
| 学习记录保存失败 | 保存面板内部显示错误，未保存内容保持可编辑 |
| Agent 请求失败 | Agent 面板内部显示错误，已有会话和题解保持可用 |

错误信息不显示堆栈、原始响应或内部实现细节。

## 可访问性

- 所有交互控件支持键盘操作。
- `:focus-visible` 使用高对比青色轮廓。
- 文字与背景对比度满足 WCAG AA。
- 图标按钮必须提供可访问名称。
- 状态变化使用适当的 `aria-live` 或 `role="alert"`。
- 底部导航不覆盖正文或系统安全区域。
- 标签切换提供 `aria-selected` 或等效状态。
- 不能只依赖颜色区分难度、掌握状态或工具结果。

### 静态检查

- `npm run build` 成功。
- TypeScript 无类型错误。
- 不新增未使用依赖。
- 页面没有横向溢出。

### 浏览器验证

在 `1280px`、`768px` 和 `390px` 三档视口检查：

1. 首页、开始解题、刷题记录、题型归纳、题目详情和 404 页面。
2. 桌面侧栏、平板紧凑栏和移动底部导航。
3. 当前路由、键盘焦点和导航跳转。
4. 加载、空数据、错误和成功状态。
5. 题目输入校验与提交状态。
6. 历史记录和题型选择。
7. 学习状态与个人备注保存。
8. Tutor Agent 消息发送、工具轨迹和确认操作。
9. 移动端题解与导师标签切换。
10. `prefers-reduced-motion` 下的界面行为。

### 回归边界

- 所有现有 API 请求路径保持不变。
- 所有现有路由保持不变。
- DeepSeek 与 Mock Provider 行为不变。
- SQLite 保存、掌握状态、个人备注和 Agent 会话行为不变。
- 未跟踪的 `deepresearch/` 目录不属于本次工作范围。

## 完成标准

- 全站使用统一的固定深色 IDE 视觉系统。
- 桌面、平板和移动导航均符合设计且无换行或横向溢出。
- 首页能够展示真实产品能力和已有学习数据。
- 解题、历史、题型和详情页在视觉与交互状态上保持一致。
- Tutor Agent 在桌面和移动端均保持可用。
- 所有加载、空数据、错误、保存和确认状态具有明确反馈。
- 不修改后端 API、数据库模型和现有业务能力。
- 前端构建成功，并通过三档视口的浏览器验收。
